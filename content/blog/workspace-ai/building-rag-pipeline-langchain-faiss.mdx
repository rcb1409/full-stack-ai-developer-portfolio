When building workspaceAI, the central challenge wasn't the LLM itself — it was getting retrieval right. Anyone can call `openai.chat.completions.create` and stream tokens. The hard part is ensuring the model only answers from *your documents*, without hallucinating facts from its training data.

This post covers the exact architecture I used: chunking strategy, embedding choice, FAISS indexing, and the LangChain chain that ties it together.

## Why RAG over fine-tuning?

Fine-tuning teaches the model *how* to answer, not *what* to answer from. If your knowledge base changes — new documents uploaded, old ones deleted — you'd need to re-fine-tune. RAG decouples the knowledge store from the model, making it dynamic by default.

For workspaceAI, users upload arbitrary PDFs. Fine-tuning on arbitrary user content is impossible. RAG is the only viable approach.

## Chunking strategy

The naive approach — split every 1,000 characters — produces terrible retrieval. A chunk might start mid-sentence, cutting off context. Instead I used LangChain's `RecursiveCharacterTextSplitter` with semantic-aware separators:

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,
    separators=["\n\n", "\n", ". ", " ", ""],
)

chunks = splitter.split_documents(docs)
```

The `chunk_overlap=64` is critical. It prevents the case where a question spans a chunk boundary — the overlap ensures both chunks contain enough context to be retrieved.

For a 450-page textbook, this produced **900 chunks** averaging ~400 tokens each.

## Embedding model

I benchmarked three options:

| Model | Latency (batch 100) | Quality (manual eval) |
|---|---|---|
| `text-embedding-ada-002` | 1.2s | Good |
| `text-embedding-3-small` | 0.9s | Better |
| `text-embedding-3-large` | 2.1s | Best |

For a teaching assistant context where retrieval quality matters more than throughput, I chose `text-embedding-3-small` — a good balance of cost, speed, and quality.

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
```

## FAISS indexing

FAISS (Facebook AI Similarity Search) is an in-memory vector store that supports efficient approximate nearest-neighbor search. For 900 vectors of 1,536 dimensions, a flat index is fast enough:

```python
from langchain_community.vectorstores import FAISS

vectorstore = FAISS.from_documents(chunks, embeddings)

# Persist to disk so we don't re-embed on every startup
vectorstore.save_local("vectorstore/")
```

Loading is instant:
```python
vectorstore = FAISS.load_local(
    "vectorstore/",
    embeddings,
    allow_dangerous_deserialization=True,
)
```

## The retrieval chain

LangChain's `RetrievalQA` chain handles the retrieve → augment → generate loop:

```python
from langchain.chains import RetrievalQA
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

retriever = vectorstore.as_retriever(
    search_type="mmr",          # Maximal Marginal Relevance — diversity over pure similarity
    search_kwargs={"k": 5, "fetch_k": 20},
)

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True,
    chain_type_kwargs={
        "prompt": CUSTOM_PROMPT,  # Forces grounded answers only
    },
)
```

The key configuration is `search_type="mmr"`. Pure cosine similarity retrieval often returns 5 near-identical chunks (e.g., the same paragraph repeated across chapters). MMR penalizes redundancy, returning 5 *diverse* chunks that collectively cover more context.

## Grounding prompt

The most important prompt engineering decision: explicitly forbid the model from using knowledge outside the retrieved chunks.

```python
from langchain.prompts import PromptTemplate

CUSTOM_PROMPT = PromptTemplate(
    input_variables=["context", "question"],
    template="""You are a teaching assistant. Answer the question using ONLY the context below.
If the answer is not in the context, say "I don't have enough information from the course materials to answer this."

Context:
{context}

Question: {question}

Answer:""",
)
```

Without this instruction, GPT-4 will happily answer from training data — defeating the entire purpose of RAG.

## Measured performance

After wiring this up and running 450+ student queries:

- **Retrieval latency**: &lt;100ms (FAISS flat index, 900 vectors)
- **End-to-end response**: ~1.2s (retrieval + GPT-4o-mini generation)
- **Positive feedback rate**: 86% (thumbs-up from students, up from 72% after tuning MMR parameters)
- **TA workload reduction**: 40% fewer routine Q&A questions

The biggest win wasn't retrieval speed — it was the *grounding*. Students immediately noticed the system cited specific textbook sections, which built trust in the answers.

## What I'd do differently

**Hybrid search**: Pure vector search misses exact keyword matches (e.g., a specific method name or algorithm). A BM25 + vector hybrid would improve precision for technical queries.

**Chunk metadata**: Storing the chapter/section as metadata and filtering on it before similarity search would reduce noise from semantically similar but contextually irrelevant chunks.

**Reranking**: Adding a cross-encoder reranker (e.g., `cross-encoder/ms-marco-MiniLM-L-6-v2`) after FAISS retrieval would further improve relevance at the cost of ~50ms latency.

For the next iteration of this system, hybrid search + reranking is the clear path to higher quality.
